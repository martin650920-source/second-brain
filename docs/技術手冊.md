# ai-workspace — 技術手冊

---

## 一、整體架構概念

### 比喻

把 ai-workspace 想成一個「個人 AI 作業系統」：

```
你的電腦
└── Google Drive（備份 + 跨裝置同步）
    └── ai-workspace/（唯一本體）
            ↓ setup 腳本建立捷徑（symlink）
        AI 工具啟動時自動讀取
```

**核心設計原則：**
- **一份本體** — 所有設定只寫一次，多個 AI 工具共用
- **按需載入** — 只燒必要的 token，不用的不載
- **單一事實來源 + symlink** — 專案 CLAUDE.md 也存一份在 ai-workspace，透過 git 版控同步到多主機，各主機用 symlink 反向連回專案目錄
- **四層分工** — rules（該怎麼做）/ mem（發生過什麼）/ projects（事實是什麼）/ skills（怎麼執行），各自獨立演化

---

## 二、兩層載入架構

```
第一層：全域層（每次都載）
    rules/global.md       ← 你是誰、偏好、AI 行為準則（Rule/記憶/Skill 建立規則）

        +

第二層：專案層（互動選單選定才載）
    projects/<name>.md            ← 專案事實（/init 或 /init-project-md 產生）
    rules/projects/<name>/*.md    ← 專案規則（全域 + path-scoped）
    mem/projects/<name>.md        ← 該專案累積筆記
```

**優先序（有衝突時）：專案層 > 全域層**

> 舊版的「情境層」（work/life + Secure Mode MCP 白名單）已於 v3 重構整套移除，日後有需要再重新設計。

## 三、Token 消耗說明

```
session 啟動
    ↓
CLAUDE.md（adapters，5行）  ← 固定消耗，極小
    ↓
context-loader/SKILL.md    ← 固定消耗，中型
    ↓
rules/global.md            ← 固定消耗，小
    ↓
projects/xxx.md            ← 選定專案才燒
rules/projects/xxx/*.md    ← 選定專案才燒
mem/projects/xxx.md        ← 選定專案才燒
    ↓
其他所有檔案               ← 0，永遠不自動載入
```

**設計紀律：**
- 規則只放「指令」，不放「內容」
- 內容放獨立 md，由 SKILL.md 按需讀取
- `rules/global.md` 建議控制在合理長度，內容量大到難維護時再拆分主題

## 四、MCP 架構

```
mcp/settings.json     ← 架構設定，只引用環境變數（${GITLAB_TOKEN} 等），進 Git
mcp/.env.example      ← 憑證欄位範本，進 Git
mcp/manifest.md        ← 各主機安裝狀態清單，進 Git（唯一允許自動 push 的例外）
~/.mcp.env             ← 真實憑證，使用者家目錄，不進 Git、不進 ai-workspace 目錄
```

新主機設定流程：複製 `.env.example` → `~/.mcp.env`，手動填入實際 key 值；shell 需
`source ~/.mcp.env` 才能讓 `${GITLAB_TOKEN}` 這類引用生效。

`mcp/manifest.md` 目前尚無安裝紀錄，第一次跑 `sync.sh` 時由腳本自動偵測補上。

## 五、公私分離機制（2026-07-02 起：改為完全公開，風險自負）

```
.gitignore 目前只排除：

/rules/global.local.md   ← 個人真實值覆蓋（若日後拆分出來）
/wiki/                   ← 第二階段知識庫（含個人知識）
/raw/                    ← 第二階段原始素材
系統/工具雜項（.DS_Store, *.swp, .obsidian/ 等）
```

**目前現況（跟舊設計不同，務必知道）：**
- `martin650920-source/ai-workspace` 這個 GitHub repo 目前是 **Public**
- `projects/*.md`（含各工作專案的 CLAUDE.md，例如晶片/DRM 相關的技術細節）**已全部進 git 版控、公開可見**
- 這是使用者明確要求拿掉白名單限制後的結果（原因：專案 CLAUDE.md 需要透過 git 同步到 WSL/SSH 等沒有 Google Drive 掛載的主機，白名單機制會擋住這條路徑），機密性由使用者自行把關，非本文件建議的預設安全姿態

**若要重新收緊：** 兩個方向都可行，各自取捨：
1. 把整個 repo 改成 Private（`gh repo edit ... --visibility private`）——最簡單，但骨架部分也一併不公開了
2. 拆成兩個 repo：`ai-workspace`（public 骨架）+ 另一個 private repo 專門放 `projects/`——多一層維護成本，但骨架仍可公開分享

歷史上曾發生過白名單機制因專案改名而失效、機密檔案外流到 public repo 的事故（已用 `git filter-repo` 清除歷史 + force push 處理），教訓是：**任何依賴「記得手動維護清單」的機制都會過期**，目前選擇乾脆不設限制，改為使用者自行注意每次要 commit 的內容。

## 七、Session 啟動完整流程

```
1. Claude Code 啟動
        ↓
2. 讀 ~/.claude/CLAUDE.md
   → symlink 指向 adapters/claude/CLAUDE.md
   → 內容：「去讀 SKILL.md」
        ↓
3. 讀 skills/global/context-loader/SKILL.md
        ↓
4. 讀 rules/global.md（你是誰、規則、偏好）
        ↓
5. Skill Drift 偵測（skills/global/ 是否都連結到 ~/.claude/skills/，缺的話問要不要補建）
        ↓
6. 先判斷這次是「自動觸發」（本次對話第一輪，session 剛開始）還是「手動觸發」
   （對話中使用者事後才說「載入 context」／`/context-loader`）——SessionStart hook
   只會在 session 開始時注入一次，「是不是第一輪」是可靠、agent 自己判斷得出來的訊號
        ↓
7. 判斷 cwd/CLAUDE.md 現況：
   已是 symlink 且目標存在 →
        自動觸發 → 直接解析出專案名稱，跳過選單（零打擾）
        手動觸發 → 仍然跳出選單（主動再叫一次代表想做點什麼）
   已是 symlink 但目標已被刪除（斷鍊） → 提示連結失效，進選單決定重新 Link 或 Remove
   真實檔案 or 不存在 → 跳選單：
        1. Link project
             → New（問名稱，若 cwd 有真實內容就直接保留搬進本體，
                    否則觸發 /init-project-md 分析 codebase）
             → 既有專案（cwd 有真實檔案的話會先問要不要備份，
                    實際執行 `sync.sh link-project ... [--backup|--force]`）
             → Skip（不載入，提示之後怎麼手動連）
        2. Remove project
             → 選專案 → 對話裡逐項確認要刪哪些 → 執行一次
               `sync.sh remove-project <name> [--paths=...] [--unlink-cwd]`
        ↓
8. 若有選定專案 <name> → 讀
   projects/<name>.md + rules/projects/<name>/*.md（若有）+
   mem/projects/<name>.md（若有）
        ↓
9. 輸出載入摘要，等待指令
```

## 八、未來擴展路徑

### 第二階段：知識圖譜（LLM-Wiki）

```
新增：wiki/ + raw/

工作流程：
  你丟資料到 raw/ → 叫 AI「消化這份，更新 wiki/」
  AI 產出變更建議 → 你在 Obsidian 確認 → 寫入 wiki/

wiki/ 結構：
  wiki/index.md              ← AI 維護的目錄
  wiki/log.md                ← 操作紀錄
  wiki/work/<主題>.md        ← 工作知識頁
  wiki/tech/<主題>.md        ← 技術知識頁
```

### 第三階段：維護自動化

```
新增：tools/health-check.sh（純腳本，零 token）

功能：
  - 孤立頁偵測（grep）
  - 斷鏈檢查（grep）
  - 過時頁偵測（find + mtime）
  - 知識庫大小統計

定期（每月）：
  - AI 對照 raw/ 重新驗證 wiki/ 內容
  - 封存 90 天未讀的過時頁
```

### 第四階段：搜尋引擎

```
當 wiki/ 超過 100 頁時：
  加入本地搜尋（BM25 + 向量）
  Claude Code 可透過 MCP 呼叫搜尋
  資料永不離開本機
```

### 情境層（若日後重新設計）

舊版 work/life 情境（含 Secure Mode MCP 白名單）已於這次重構移除。若未來要重新設計，
需要重新決定它跟 `rules/` 三層（global / project 全域 / project path-scoped）如何整合，
而不是照搬舊版的三層優先序（專案 > 情境 > 全域）。

### 公私分離機制重新收緊（若日後需要）

目前 `projects/*.md` 全部進 public repo（見第五節）。若日後想恢復隔離，建議採「拆成兩個 repo」
而非走回白名單 `.gitignore`——後者已證實會因專案改名/新增而失效，是本次事故的根因。

### Schema 版本化（長期）

```
當 SKILL.md 規則演化時：
  在 rules/global.md 加版號
  舊頁面 migrate 到新格式
  避免規則改了但舊知識頁還用舊格式
```
